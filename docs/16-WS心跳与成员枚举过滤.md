# 16 · WS 心跳 + 成员枚举过滤（④ B 类协议增强）

> 定位：把 backlog ④ 的两个协议子项（WS 心跳、成员枚举过滤）落成方案 + 真相源。
> Owner：aikey@rayson。共享文件 `broker.ts`（① 不碰，归我）+ `broker-client.ts`（待房主确认归属）。**不碰** room-bridge.ts / room-manager / room-service / presence（房主域）—— 成员枚举过滤只在 broker 投递/响应侧，不改 presence 数据本身。

## 1. 原始需求

**WS 心跳**：长连 broker ↔ edge 之间，**死连接探测**目前依赖 TCP/重连兜底。半开连接（对端进程没了但 socket 没 FIN）会让 broker 误以为成员还在线、继续往一个黑洞投递，也让 presence 名单含"幽灵成员"。需要应用层 ping/pong 主动探活 + 超时剔除。

**成员枚举过滤**：`room_members` / `listMembers` 当前返回房间**全部**成员。随成员增多 + 跨租户/跨机场景，需要按可见性规则过滤枚举结果（如：只列在线成员、或按调用者权限过滤）——避免无差别暴露完整名单（信息最小化）。

**为什么要**：心跳关乎**投递可靠性 + presence 准确性**（主链路鲁棒）；枚举过滤关乎**信息暴露面**（安全：中）。两者都是 broker 侧、与 ① 多 agent-type 接入正交。

## 2. 现状分析

- `broker-client.ts`：已有等抖动指数退避重连（§8.2），但**无应用层 ping/pong**——靠底层 socket 事件，半开连接探测不及时。
- `broker.ts`：成员 fanout 用 membership 数组 + MEMBER_CACHE（:16 MEMBER_CACHE_CAP）。`listMembers` 直接回全名单。
- presence（房主域）：成员在线/元数据。**本方案不改 presence 数据**，只在 broker 投递/枚举响应侧用它。

## 3. 方案

### 3.1 WS 心跳（ping/pong + 超时剔除）

- broker.ts：对每个连接维持 `lastSeen`；周期性 ping，超 `heartbeatTimeoutMs` 无 pong → 判死、关连接 + 触发 member_left（复用现有离开事件，不新增）。
- broker-client.ts：收 ping 回 pong（或反向，按实现简单择一）；心跳失败计入既有重连路径。
- 参数可配（`heartbeatIntervalMs`/`heartbeatTimeoutMs`），默认值保守（如 30s/75s），避免误杀慢网络。

### 3.2 成员枚举过滤

- broker.ts `listMembers` 响应侧加**过滤钩子**：默认按"在线 + 调用者可见"过滤。规则用**配置表/枚举**穷举可见性档位，不写 if-else 补丁（遵循"配置表而非胶水逻辑"）。
- MVP 先实现"只列在线成员"（依赖 §3.1 心跳的在线判定），权限档位过滤留接口、配置驱动。

### 3.2.1 为什么过滤放 broker 而非收端
信息最小化是安全属性，必须服务端收窄——收端过滤等于全名单已发出（同 DM 私聊的安全论证）。

## 4. 影响文件 + 精确插入点

| 文件 | 归属 | 改动 | 插入点 |
|---|---|---|---|
| `src/broker.ts` | 共享，① 不碰，归我 | ping 调度 + lastSeen + 超时剔除；listMembers 过滤钩子 | WS message/open/close 处理器；listMembers 响应构造处 |
| `src/broker-client.ts` | 待房主确认 | 收 ping 回 pong | message 处理器 |
| presence / room-service | 房主域 | **零改动**（只读在线/成员数据） | — |

> broker.ts 心跳调度 = 新增独立函数 + 一个 setInterval，**和 ③ DM 的 to[] 收窄投递分支不同区域**（DM 在 fanout 分支，心跳在连接生命周期），自洽不撞——两者同属我 owner，内部排序即可。

## 5. 分阶段实施

- **P1**：WS 心跳 ping/pong + 超时剔除 + member_left 触发。验收：杀掉一个 edge 进程（不优雅关）→ broker 在 timeout 内判死、其余成员收到 member_left；围栏测试用可注入时钟。
- **P2**：listMembers"只列在线"过滤（依赖 P1 在线判定）。验收：离线成员不出现在 room_members。
- **P3**：可见性档位配置表（留扩展，MVP 可不实现）。

## 6. 风险与边界

- 心跳超时**宁松勿误杀**：默认 timeout 给慢网络留余量；误杀活跃成员比晚剔除幽灵更糟。
- 过滤规则走**配置表穷举**，不堆 if-else。
- 不改 presence 数据模型 / room-service（房主域）；只在 broker 投递与枚举响应侧消费。
- 与 ① 正交，可在 ① 之前/并行实现（broker.ts 在 ① 不被碰）。
