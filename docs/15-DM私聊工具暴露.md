# 15 · DM 私聊工具暴露（envelope to[] 已支持，工具面未暴露）

> 定位：把 backlog ③「DM 私聊」从"协议已支持、工具面没暴露"落成方案 + 真相源。
> Owner：aikey@rayson。独占文件：`src/claude-adapter.ts`。共享文件：`broker.ts`（to[] 定向投递，① 不碰 broker.ts，归我）。**不碰** `room-bridge.ts`/`renderRoomEvent`（房主单主）。

## 1. 原始需求（What + Why）

**现状**：`room_say` 的 `to=[ids]` 只是"@提及高亮"——消息**仍广播给房间每个成员**（claude-adapter.ts:562 工具描述明说"Everyone still receives the message"）。Envelope 已有 `to[]` 字段（房主确认"to[] 已支持"），但**没有一条"只发给 to[]、别人收不到"的私聊通路**。

**为什么要**：多人/多 agent 房间里，有些协调是**点对点**的（"我俩先对齐这个接口再广播结论"），不该刷屏整个房间，也不该让无关成员看到半成品讨论。今天只能靠广播 + @，等于"当众私聊"。

**目标**：暴露一个 **DM 私聊工具**——消息只投递给指定成员，房间其他人收不到。复用已有的 envelope `to[]` 语义，不新增协议字段。

## 2. 现状分析

- **工具面**（claude-adapter.ts）：只有 `room_say`(广播) + `room_members`。`room_say` 的 `to[]` → 提及高亮，不改变投递范围。
- **协议**（envelope）：`to[]` 字段已存在。
- **broker 投递**（broker.ts）：当前对房间消息**扇出给所有成员**（fanout），不读 `to[]` 做定向收窄。

所以 gap 只在：①一个表达"这是私聊"的工具入口；②broker 对"私聊型 envelope"按 `to[]` **收窄投递**而非扇出。

## 3. 方案（复用 to[]，新增定向投递语义）

### 3.1 核心思路

- 新增工具 `room_dm`（claude-adapter，我独占）：参数 `to=[ids]`（必填，≥1）+ `text`。语义="仅这些成员可见"。
- Envelope 复用现有 `to[]`，新增一个**投递模式标记**区分"广播+提及"vs"私聊定向"——用 envelope 已有元数据承载（如 `delivery:"direct"`），**不新增顶层协议字段**（careful-api-creation：优先 metadata/enum variant）。
- broker.ts：投递时若 `delivery==="direct"`，**只发给 `to[]` ∩ 成员**（且发件人必须是成员、to[] 必须都是成员，否则拒；非成员一律收不到）。否则维持现状扇出。

### 3.2 为什么这样设计（对比方案）

| 方案 | 新增独立 DM 协议/topic | **复用 to[] + delivery 标记（选）** | 客户端过滤（broker 仍广播，收端丢弃） |
|---|---|---|---|
| 协议表面积 | ❌ 新 topic/字段 | ✅ 复用 to[]，加一个 enum 标记 | 0 |
| 隐私 | 强 | 强（broker 端收窄，非目标永不投递） | ❌ 假私聊（消息仍到所有人机器，靠收端自觉丢） |
| 实现成本 | 高 | 中 | 低但**不安全** |

选中间方案：隐私由 **broker 端收窄投递**保证（非目标成员根本收不到字节），不是收端自觉——这是安全底线（"客户端过滤"方案直接否掉，等于把私聊内容发到所有人机器）。复用 to[] 避免协议膨胀。

### 3.3 渲染（不碰 renderRoomEvent）

DM 到达收端后，复用房主 `renderRoomEvent` 现成的 **"@你"定向渲染**（room_say 分支已有 `tag` 逻辑）。若要"🔒 私聊"独立视觉，由房主在 renderRoomEvent 加一行（数据来自 env.to[]，我负责路由进来）——**我不碰该函数**（边界：room-bridge.ts 单主归房主）。

## 5. 分阶段实施

- **P1**：envelope `delivery` 字段 + broker.ts 定向收窄投递 + 围栏测试（非目标成员收不到、to[] 含非成员被拒）。**前置依赖**：① 多 agent-type 进房间落地（否则只有 Claude 间 DM，价值有限）。
- **P2**：claude-adapter `room_dm` 工具暴露 + 工具描述（明确"仅 to[] 可见"）。
- **P3**：（可选，房主）renderRoomEvent 加"🔒 私聊"视觉。

## 6. 风险与边界

- **隐私是安全底线**：定向投递必须 broker 端收窄，非目标成员永不收到字节。围栏测试必须断言"非目标成员的 onEvent 没收到该 DM"。
- 发件人 + 所有 to[] 必须是房间成员，否则拒（复用现有 isMember 校验）。
- 不碰 renderRoomEvent / room-bridge.ts（房主单主）；不新增顶层协议字段（复用 to[] + 可选 delivery 标记）。
- 依赖 ① 落地才有多 agent-type DM 的完整价值（依赖排序：① 先）。
