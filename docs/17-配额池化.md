# 17 · 配额池化（④ B 类，含待决策开放项）

> 定位：把 backlog ④「配额池化」框定为需求 + 真相源。**B 类、低优先**，且**核心范围有歧义、需拍板**——本文先框定问题与候选方案，不预设实现，待决策后再补细节。
> Owner：aikey@rayson。独占文件：`src/budget/*`。**不碰** broker/room 协议域。

## 1. 背景：现状已有什么

`src/budget/` 已实现**同机跨 pair 配额协调**：同一台机器上每个 bridge pair 看到的是**同一个账号配额**（`advice-cooldown.ts`:5 注释"every bridge pair on this machine sees the SAME account quota"），所以有 `BudgetCoordinator` / `AdviceCooldown` 防止多 pair 各自重复消耗/重复告警。窗口含 5h + weekly（`quota-source.ts` / `budget-coordinator.ts`）。

**所以"同机跨 pair 池化"已经做了。** backlog ④"配额池化"指的应是**更大范围**。

## 2. 🔴 待决策：池化范围是什么？（需拍板）

"配额池化"可能指三种之一，**实现复杂度、价值、风险差异极大**，必须先定：

| 候选 | 含义 | 价值 | 复杂度/风险 |
|---|---|---|---|
| **A. 跨机/房间级配额可见** | room 内成员互相**看到**彼此配额水位（只读聚合），协调"谁该歇会儿"。复用现有 `get_budget` + 房间事件广播配额快照 | 中：团队避免同时撞软线 | 低-中：只读聚合 + 广播，无强一致 |
| **B. 跨机团队配额池（共享额度账本）** | 多成员**共用一个额度池**，强一致扣减，超额阻断 | 高：真正的团队额度管理 | **高**：分布式额度账本 + 强一致 + 争用，违背"无状态优先/可追溯即可"的成本优先级 |
| **C. 仅审计聚合** | 把各成员配额事件汇到可追溯账本（CQRS 旁路），不做实时协调，后续按需衍生 | 低-中：先有数据 | 低：旁路追加，不碰主链路 |

**我的倾向：A 或 C，否决 B（至少本期）**。理由（按成本优先级表）：B 引入分布式强一致额度账本——稳定性风险 + 维护成本 + 不可演进风险都高，ROI 低；A 复用现有 `get_budget` + 房间广播（和 ① 多 agent-type/presence 天然契合），增量小、可演进；C 是最保守的"先存可追溯数据"。

## 3. 方案骨架（取决于 §2 决策）

- **若选 A**：每成员周期性把自己的配额快照（5h/weekly %）作为一种房间事件广播（复用 envelope，类似 task_completed）；收端聚合渲染"房间配额面板"。`budget/` 加快照导出 + 房间事件编解码；**不碰** broker 投递语义（走现有事件通路）。
- **若选 C**：`budget/` 把配额事件写入本地可追溯账本（CQRS 旁路），不实时协调。
- **若选 B**：需单独立项做分布式额度账本设计——本文不展开（建议否决）。

## 4. 影响文件（A 方案预估）

| 文件 | 改动 |
|---|---|
| `src/budget/quota-source.ts` / `budget-state.ts` | 导出配额快照（只读） |
| `src/budget/budget-coordinator.ts` | 周期触发快照广播 |
| envelope / 房间事件编解码 | 新增 `budget_snapshot` 事件类型（复用 envelope，不新增 topic） |

> 与房主域的交叉：配额快照若走房间事件广播，渲染会落到 `renderRoomEvent`（房主单主）——届时**由房主加渲染分支**，我只负责 budget 侧的快照导出 + 事件发布。需 cross-check。

## 5. 分阶段实施

**先决条件 = §2 拍板**。拍板前不写实现。
- 选 A：P1 配额快照导出（只读，可独立测）→ P2 房间事件广播 → P3 房主侧聚合渲染。
- 选 C：P1 配额事件落本地账本（旁路，不碰主链路）。

## 6. 风险与边界

- **否决 B（本期）**：分布式强一致额度池的稳定性/维护/演进成本 > 价值，违背成本优先级表。要做需单独立项。
- A 方案若走房间事件，渲染归房主（renderRoomEvent 单主）——又一处需 cross-check 的交叉点。
- B 类低优先：① Codex 接入、⑥ 非破坏部署落地后再排。
